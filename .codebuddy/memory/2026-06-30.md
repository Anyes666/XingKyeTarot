### AvatarCropPage 手势失效修复（19:18）

用户反馈：上传图片后,圆形裁剪框内没法用手指放大/缩小/移动图片。

**根因分析**：
1. `gesture` 绑定在独立的空透明 Column 上,跟 Image 是 Stack 中两个独立兄弟节点,hitTest 命中不可靠（尤其在嵌套 Stack 场景下）
2. `handlePinchUpdate` 每次缩放都把 offset 重置到中心,导致用户先拖动再缩放时图片位置跳变
3. 装饰用金色边框 Stack 覆盖在图片之上,可能截获 hitTest
4. 顶部栏 Row 默认 hitTest 行为可能遮挡下方手势层

**修复方案**：
- 把 `gesture()` 直接绑到包 Image 的 Column 上,确保手势和图片在同一组件层
- 缩放时不再重置 offset,只做 clamp 限制(用户拖动位置得到保留)
- 装饰金色边框 Stack 加 `.hitTestBehavior(HitTestMode.None)` 透传
- 删除多余的空透明 Column 占位
- 遮罩层保持 `.hitTestBehavior(HitTestMode.None)`

**编译结果**：BUILD SUCCESSFUL in 509 ms, ERROR=0

### 模拟器手势仍不生效 — 第三轮修复（19:27）

用户反馈：在模拟器里仍然无法拖动和缩放图片。

**进一步根因分析**：
`.clip(true)` + `.borderRadius()` 的组合在某些 ArkUI 版本/模拟器环境中会**阻止 gesture 事件透传**，因为 clip 改变了 HitTest 行为——这不是代码逻辑错误，而是渲染层行为差异。

**第三轮修复方案**：
- **图片容器**（Column）：只负责显示，加 `.hitTestBehavior(HitTestMode.None)` 让它不参与触摸事件
- **手势 overlay**（独立 Column）：透明、无 clip/border、`.hitTestBehavior(HitTestMode.Default)`、`.zIndex(998)`（仅低于 Toast 的 999）——保证在所有元素最上层接收触摸
- Image 改回 `.offset()` 替代 `.position()`（Column 内不支持 position）
- `clampOffset` 简化为直接 min/max 边界计算（适配直接 width/height 模型而非 scale/offset 模型）
- `handlePinchUpdate` 改为围绕圆心缩放（保持圆心处图片像素不变）
