## 2. 角色机制分析

### 2.1 角色如何出现

- **无固定角色文件**。"对方"是一个可由用户任意改名的占位（`settings.partnerName`，默认"对方"/"梦角"），头像、昵称、状态签名全部用户上传。
- 角色存在感完全靠**拟真聊天剧场**堆出来：头像 + 昵称 + 状态 + "正在输入" + 已读回执 + 延迟回复 + 表情贴纸点缀。
- 没有任何角色画像、性格设定、语气规范。角色的"性格"=用户自建回复库的语气。

### 2.2 角色如何说话（核心机制）

`simulateReply()` 的真实逻辑（**这是本项目最重要的一处**）：

```
用户发消息后：
1. 1.5~4s 内把用户消息标为"已读"
2. 概率判断："已读不回"几率（allowReadNoReplyChance）
3. 若要回复：
   a. 显示"X 正在输入"指示器
   b. 随机延迟 replyDelayMin~replyDelayMax
   c. 回复条数：75%→1条 / 20%→2条 / 5%→3条
   d. 从 customReplies 池随机抽（最多重试6次找非空）
   e. 20% 概率前/后拼一个自定义 emoji
   f. 20% 概率追加一张贴纸（延迟 0.4~1s）
   g. 30% 概率"引用回复"用户最近一条消息
4. 3% 概率不回文字，改为"拍一拍"互动
5. 小概率切换 persona（头像+昵称）
```

**关键结论：对方的回复没有任何理解、没有上下文、没有 AI，是纯概率 + 拟真时序 + 池随机。** 陪伴感来自"剧场效果"，不是智能。

### 2.3 是否有固定语气 / 状态变化 / 情绪回应

| 维度 | 情况 |
|---|---|
| 固定语气 | 无。语气=用户自建回复库的语气 |
| 状态变化 | 有。partner 状态签名可变；persona 随机切换（头像+昵称） |
| 情绪回应 | 无真正情绪识别。`mood.js` 有心情日历，但 partner 心情是**随机生成**的（20% 几率当天不生成），不是基于对话 |
| 用户关系推进 | 无好感度、无剧情进度。关系=用户脑补 |
| 新手引导职责 | 否。教程由独立 tour 系统承担，不是角色带领 |

### 2.4 对星钥塔罗的适配判断

| milk 做法 | 适合谁 | 说明 |
|---|---|---|
| 拟真时序（typing/delay/已读） | ⭐ 星澜 | 星澜点击后可加"正在输入" + 短延迟，增强陪伴感 |
| 回复池随机抽取 | ⭐ 星澜 | 星澜文案库按场景分组随机，离线即可 |
| 表情/贴纸点缀 | ⚠️ 谨慎 | 星钥塔罗气质克制，emoji 要极少或不用 |
| 引用用户消息回复 | ❌ 不适合 | 星钥塔罗不是聊天，是情绪陪伴，无需"引用" |
| persona 随机切换 | ❌ 不适合 | 星澜人设必须稳定，不能随机变 |
| "已读不回"几率 | ❌ 不适合 | 星钥塔罗是工具型，星澜应及时回应 |
| partner 心情随机生成 | ⚠️ 改造 | 可改造为"星澜今日状态"（如月相），但不应模拟情绪回应 |
| 用户自建回复库 | ⚠️ 部分 | 星钥塔罗 v1 不必开放用户自建文案，但可预留 |

**星衡的启示**：milk 没有"理性引导者"角色。星衡的"理清方向"是星钥塔罗独创，milk 无直接借鉴，但 milk 的"拟真时序"同样可用于星衡（星衡"正在梳理" + 延迟 + 结构化输出）。

## 3. 对话数据结构分析

### 3.1 文本存储方式

| 数据 | 存储形式 | 位置 |
|---|---|---|
| 内置回复 | `string[]`（本版为空 `REPLY_MESSAGES: []`） | `config.js` CONSTANTS |
| 用户自建回复 | `string[]`（纯字符串数组，无元数据） | 全局 `customReplies` |
| 回复分组 | `{name, items[], disabled}[]` | `window.customReplyGroups` |
| 消息记录 | `object[]`（id/sender/text/timestamp/image/status/favorited/note/replyTo/type） | 全局 `messages` |
| 拍一拍动作 | `string[]` | `customPokes` |
| 状态签名 | `string[]` | `customStatuses` |
| 格言 | `string[]` | `customMottos` |
| 欢迎语 | `{line1, line2}[]`（90+ 条诗意短语） | `config.js` WELCOME_ANIMATIONS |

### 3.2 能力矩阵

| 能力 | 是否支持 | 实现方式 |
|---|---|---|
| 分支对话 | ❌ | 无分支节点，纯线性 + 随机 |
| 多轮上下文 | ❌ | 不记忆对话内容，仅 30% 概率引用最近 10 条中的随机一条 |
| 随机回复 | ✅ | `Math.random()` 从池抽取 |
| 条件触发 | ⚠️ 仅概率 | "已读不回"几率、拍一拍几率、贴纸几率，非内容条件 |
| 历史记录 | ✅ | `messages[]` 持久化到 localforage |
| 首次进入专用文案 | ✅ | splash 6 屏 + disclaimer + tour |
| 教程专用对话 | ❌ | 教程是独立浮层，不融入对话流 |
| "已看过教程不再显示" | ✅ | `localforage tour_seen` |
| 跳过教程 | ✅ | tourSkipBtn |
| 重新查看教程 | ✅ | 数据管理页"重放新手引导"按钮 |

### 3.3 迁移 ArkTS 的适配性

milk 的数据结构**几乎全部不符合 ArkTS 严格类型**：
- `customReplies: string[]` 裸字符串数组 → 可直接迁移（`string[]` 合法）
- `messages: object[]` 匿名对象数组 → **必须**改 `MessageItem[]` + 显式 `interface MessageItem`
- `DOMElements` 嵌套对象字面量 → **必须**拆成 `@State` / `@Ref`，ArkTS 无 DOM
- `settings: {}` 空对象 → **必须**显式 `interface AppSettings`

**结构建议（仅建议，不写代码）**：milk 的"池随机 + 概率装饰"模式可迁移，但数据必须显式 interface 化。星钥塔罗的文案库应比 milk 更结构化（见第 11 节）。

## 4. 用户选择与流程分析

### 4.1 推进方式

| 方式 | 用途 |
|---|---|
| 输入框 + 发送按钮 | 主交互，发文字/图片 |
| 点击头像/昵称/状态 | 触发编辑 modal |
| 点击消息 | 弹出操作菜单（收藏/回复/注释/删除） |
| 长按消息 | 同上 |
| 底部功能按钮 | 拍一拍、继续说、批量、附件、设置、收藏、会话 |
| tour 中的"下一步/上一步/跳过" | 教程推进 |
| splash 的"上一页/下一页/进入" | 门禁推进 |

### 4.2 状态变量与节点

- **无剧情节点、无好感度、无情绪值、无进度值**。这是工具型而非养成型。
- 状态变量全是"配置型"：`isBatchMode`、`currentReplyTo`、`displayedMessageCount`、`currentTourStep`、`currentMoodPage` 等。
- 阶段性反馈：运势占卜的周运/日运结果（一次性展示）、纪念日动画。

### 4.3 教程阶段与操作步骤

tour 22 步覆盖：头像→昵称→状态→对方头像/昵称→格言→输入框→发送→附件→拍一拍→继续说→批量→设置→外观设置→聊天设置→高级功能→数据管理→日夜切换→收藏→会话→消息操作→完成。**每个功能逐一高亮讲解**。

### 4.4 对星钥塔罗的流程借鉴

| milk 流程 | 是否借鉴到星钥塔罗 | 改造位置 |
|---|---|---|
| 输入框发消息 | ❌ | 星钥塔罗不是聊天 App，无输入框 |
| 点击角色触发反馈 | ⭐ 借鉴 | 首页点击星澜 → 星澜说一句（从文案库随机） |
| 拍一拍互动 | ⚠️ 改造 | 可改"轻触星澜" → 星澜轻声回应 |
| tour 高亮逐步教学 | ⚠️ 大幅精简 | 星钥塔罗最多 3-5 步，且融入对话流而非独立浮层 |

| 批量发送 | ❌ | 不适用 |
| 运势占卜抽牌 | ⚠️ 仅交互 | 3D 翻牌动画可借鉴，"运势"框架必须避开 |

## 5. 新手介绍与操作引导分析（重点）

### 5.1 milk 的三层门禁式新手流程

```
启动 → loading 进度条（"正在建立安全连接..."等假进度文案）
     → splash 6 屏承诺门禁（逐字抄写承诺才放行，localStorage splashPledgeSigned_v3）
     → 免责弹窗（关于&声明：致谢 + 非商用声明，tour_seen 未设时弹出）
     → 接受免责 → 启动 22 步 tour（高亮框 + 浮层，prev/next/skip）
     → tour 完成 → 每日问候弹窗（daily-greeting-modal）
     → 进入主界面
```

### 5.2 tour 系统的技术结构（值得借鉴的数据结构）

```js
tourSteps = [
  { title, content, position: 'center' },                    // 居中无目标
  { element: '#my-avatar', title, content, position: 'bottom', onBefore: ()=>{} }, // 高亮目标
  ...
]
// position: top/bottom/left/right/center
// onBefore: 钩子，可在展示前打开/关闭 modal 以暴露隐藏元素
// 高亮框 = 目标元素 getBoundingClientRect + 10px padding
// 浮层位置 = 按 position 计算 + 视口边界 clamp
// 持久化：localforage tour_seen = true
// 重放：数据管理页"重放新手引导"按钮 → removeItem('tour_seen') → startTour()
```

### 5.3 评估

| 维度 | 评价 |
|---|---|
| 首次进入介绍 | ✅ 有，但 6 屏承诺门禁是社区防御机制，与星钥塔罗无关 |
| 角色带领介绍 | ❌ 无，教程是机械浮层，非角色口吻 |
| 逐步引导 | ✅ 22 步，**过长** |
| 功能/按钮/页面说明 | ✅ 每步讲一个元素 |
| 弹窗/浮层/高亮 | ✅ 高亮框 + popover 浮层 + 遮罩 |
| 跳过教程 | ✅ tourSkipBtn |
| 重新查看 | ✅ 数据管理页重放按钮 |
| 记录完成状态 | ✅ tour_seen |
| 教程长度 | ❌ **22 步严重过长**，移动端劝退 |
| 是否打断用户 | ❌ 强制走完才进主界面（接受免责后才 tour） |
| 自然融入主流程 | ❌ 独立于主流程的"新手村" |
| 移动端适配 | ⚠️ 浮层会遮挡，position 计算在小屏偶有溢出 |

### 5.4 星钥塔罗可借鉴的新手引导方案草案（仅设计，不改代码）

**核心原则**：不用独立浮层教程，**把引导融进星澜的对话流**。星澜轻轻说，用户轻轻点。

#### ① 首页首次引导
- 触发：`isFirstLaunch === true`
- 形式：首页星澜直接显示一句（非弹窗）：
  > "你来了。不急，先坐坐。"
- 下方出现一个克制的选项条（非按钮组，是星澜话后的轻选项）：
  - "第一次来，走走看看" → 进入②
  - "我想直接抽一张" → 跳到"开始今日一占"，但星澜补一句："好。无论抽到什么，都只是一个入口。"
- 完成后写 `hasSeenHomeGuide = true`

#### ② 单牌首次引导
- 触发：首次进入单牌页
- 星澜一句："一张牌，一个此刻的映照。想着你现在最贴近的感受就好。"
- 不解释牌意机制，只在抽牌前给一句安抚（见④）

#### ③ 圣三角首次引导
- 触发：首次进入圣三角
- 星澜一句："过去、现在、未来——不是预言，是三个角度看自己。"
- 不强调"未来"，避免预言感

#### ④ 关系探索首次引导 + 抽牌前安抚
- 星澜："不用想清楚要问什么。先选一个最贴近现在的感受。"
- 情绪选项（复用现有 EmotionTag）作为入口
- 抽牌动画前一句："深呼吸。这张牌不是答案，是镜子。"

#### ⑤ 结果页首次引导
- 结果出现后，星澜在底部轻声：
  > "这不是答案，只是一个自我探索的入口。如果觉得不对，也没关系。"
- 首次显示"再听星澜说一句"按钮（即 milk 的"继续说"改造）
- 底部固定一行小字免责声明（已存在，保留）

#### ⑥ 记录页首次引导
- 首次进入记录页，星澜一句：
  > "这里留着之前的自己。想看就看，不想看也没关系。"
- 不强调"回顾反思"，避免说教

#### ⑦ 免责声明温和提示
- 不用弹窗，用首页星澜对话流里一句：
  > "这里的一切，只是陪你看看自己。不预言，不建议。"
- 首次显示后 `hasSeenDisclaimerHint = true`，不再主动弹

#### ⑧ 跳过与重看
- 每个引导步底部有一个极淡的"跳过"（不强制走完）
- 设置页加一个"重新听听星澜的介绍"入口（对应 milk 的"重放教程"），但触发的是星澜对话流而非浮层

## 6. 情绪陪伴设计分析

### 6.1 milk 的情绪机制

| 机制 | 内容 | 评估 |
|---|---|---|
| 心情日历（mood.js） | 12 固定心情 {key,kaomoji,label,color} + 自定义 | ⚠️ 双视角（我/对方），关系框架 |
| partner 心情 | 每天随机生成（20% 几率不生成），partnerNote 从 customReplies 随机 | ❌ 假的情绪回应，星钥塔罗不可用 |
| 安慰话术 | 无专门安慰库，靠用户自建回复 | 无可借鉴话术 |
| 倾听话术 | 无 | 无 |
| 鼓励话术 | 无 | 无 |
| 引导表达 | mood 日历的"随记"输入框 | ⚠️ 可借鉴"今日小记" |
| 教程中照顾情绪 | ❌ 教程是机械说明 | 无 |
| 操作失败反馈 | `showNotification('...', 'error')` toast | ⚠️ 过于生硬，星钥塔罗应更柔 |

### 6.2 风险表达（milk 中存在的、星钥塔罗必须避开的）

| milk 表达 | 风险 |
|---|---|
| WELCOME_ANIMATIONS 大量"命运的红线""灵魂共振""你是我的光" | 恋爱化 / 宿命化 |
| "运势占卜"框架（周运/日运） | 算命承诺 |
| partner 随机心情伪装成"对方今天的心情" | 虚假情绪回应 |
| "拍一拍"等亲密互动 | 恋爱养成感 |

### 6.3 可改写成星澜风格的表达方向

milk **本身没有可借鉴的安慰话术**（它依赖用户自建）。但对星澜的启发是：

- 星澜的安慰 = **接住 + 不评价 + 不承诺**。如："嗯，我在。" / "听起来不容易。" / "不用现在就想明白。"
- 星澜的鼓励 = **轻推 + 留退路**。如："试试看？不想也没关系。"
- 星澜的引导表达 = **给选项而非追问**。如："先选一个最贴近现在的感受。"（复用 EmotionTag）
- **轻提示插入点**：抽牌前、结果出现后、记录页空状态。**不应打扰的点**：用户阅读结果时、用户在设置页时。

## 8. 本地离线实现价值

### 8.1 milk 的离线实现

| 维度 | 情况 |
|---|---|
| 不联网运行 | ✅ 完全本地，无任何网络请求（除可选 Notification API） |
| 依赖外部接口 | ❌ 无 |
| 本地数据 | ✅ localforage（IndexedDB）存大对象 + localStorage 存标志位 |
| localStorage 用法 | splashPledgeSigned_v3 / notifEnabled / dailyGreetingShown / disabledReplyItems / exportReminderLastShown / BACKUP_V1_critical |
| localforage 用法 | sessionList / chatMessages / settings / partnerAvatar / tour_seen / moodCalendar / weekly_fortune / daily_fortune_3 / annHeaderBg_xxx 等 |
| 紧急备份 | visibilitychange/pagehide/beforeunload 触发 `_backupCriticalData()` 存 localStorage |
| 自动恢复 | 启动时 `_tryRecoverFromBackup()` 检测备份比当前多则恢复 |

### 8.2 迁移星钥塔罗的判断

| milk 机制 | 迁移价值 |
|---|---|
| localforage 分键存储 | ⭐ → ArkTS PreferencesHelper / 关系型数据库 |
| 标志位 localStorage | ⭐ → PreferencesHelper 存 isFirstLaunch 等 |
| 紧急备份 + 自动恢复 | ⚠️ 星钥塔罗数据量小，可不做，但"定时保存"思路可借鉴 |
| tour_seen 持久化 | ⭐ → `hasSeenXxxGuide` 一组布尔 |
| 不联网 | ⭐ 星钥塔罗本就离线，天然契合 |
| Notification API | ❌ 星钥塔罗不推送，不用 |

**结论：milk 的离线架构与星钥塔罗高度契合，标志位管理可直接借鉴。**

## 11. ArkTS 结构建议（仅结构，不改代码）

### 11.1 角色文案数据结构

```typescript
// 建议：比 milk 的 string[] 更结构化
interface CompanionLine {
  id: string;                    // 唯一标识
  scene: CompanionScene;         // 场景枚举
  text: string;                  // 文案内容
  tone: CompanionTone;           // 调性枚举
}
type CompanionScene = 'HOME_TAP' | 'BEFORE_DRAW' | 'RESULT_OPEN' | 'RESULT_REASSURE' | 'RECORD_EMPTY' | 'GUIDE_FIRST';
type CompanionTone = 'GENTLE' | 'NEUTRAL' | 'REASSURE';
```
milk 的 `customReplies: string[]` 升级为带场景/调性的结构，便于"按场景抽 + 按调性筛"。

### 11.2 对话节点数据结构（轻量，不做剧情）

```typescript
interface CompanionTurn {
  lineId: string;
  text: string;
  options: CompanionOption[];     // 星澜话后的轻选项，可空
}
interface CompanionOption {
  label: string;
  nextLineId: string;             // 下一句，或 'END'
}
```
**不做 milk 没有的分支树**，只做"星澜说一句 + 0~2 个轻选项 + 选项指向下一句"。

### 11.3 用户选择数据结构

```typescript
interface EmotionChoice {
  tag: EmotionTag;                // 复用现有 6 标签
  timestamp: number;
}
```
不引入 milk 的 12 心情 + 自定义，保持现有 6 标签克制。

### 11.4 新手引导步骤数据结构

```typescript
interface GuideStep {
  stepId: string;
  scene: GuideScene;              // HOME / SINGLE / TRIANGLE / RELATION / RESULT / RECORD
  starLanLine: string;            // 星澜说的话（替代 milk 的 title/content）
  options: GuideOption[];         // 轻选项
  canSkip: boolean;
}
interface GuideOption {
  label: string;
  action: 'CONTINUE' | 'JUMP_TO_DRAW' | 'END';
}
type GuideScene = 'HOME_FIRST' | 'SINGLE_FIRST' | 'TRIANGLE_FIRST' | 'RELATION_FIRST' | 'RESULT_FIRST' | 'RECORD_FIRST';
```
**关键差异**：milk 的 `element/position/onBefore`（高亮浮层）全部去掉，改为 `starLanLine`（星澜口吻）。

### 11.5 本地状态数据结构

```typescript
interface OnboardingFlags {
  isFirstLaunch: boolean;
  hasSeenHomeGuide: boolean;
  hasSeenSingleGuide: boolean;
  hasSeenTriangleGuide: boolean;
  hasSeenRelationGuide: boolean;
  hasSeenResultGuide: boolean;
  hasSeenRecordGuide: boolean;
  hasSeenDisclaimerHint: boolean;
}
```
对应 milk 的 `tour_seen`，但粒度更细（每个场景独立标志）。用 PreferencesHelper 存。

### 11.6 首页星澜点击文案库扩展建议

- 现有：点击星澜出文案（已有机制）
- 借鉴 milk：增加**拟真时序**（"星澜正在输入" + 0.8~1.5s 延迟）
- 借鉴 milk：文案按场景分组随机（HOME_TAP 池）
- **不借鉴**：milk 的 emoji/贴纸点缀、引用回复

### 11.7 星衡未来对话系统扩展建议

- milk 无对应角色，星衡是星钥塔罗独创
- 星衡可用同样的"拟真时序"（"星衡正在梳理" + 稍长延迟，体现"理性"）
- 星衡文案调性 = NEUTRAL / STRUCTURED，选项更偏"理清"而非"接住"
- **不做** milk 式的 persona 切换

### 11.8 保持轻量的原则

- 不做 milk 的 22 步教程 → 最多 3-5 步对话流
- 不做 milk 的心情日历双视角 → 保留现有 7 日情绪星图
- 不做 milk 的 customReplies 用户自建 → v1 内置文案库
- 不做 milk 的运势占卜 → 塔罗只做"映照"不做"运势"
- 不做 milk 的多会话/群聊/信封/通话 → 星钥塔罗聚焦核心

## 12. 星钥塔罗的新手引导改造建议（单独章节）

### 12.1 第一次打开首页时，星澜可以说什么

> "你来了。不急，先坐坐。"

随后底部出现两个极淡选项（非弹窗）：
- "第一次来，走走看看" → 星澜："好，我陪你看看。"（进入轻引导）
- "我想直接抽一张" → 星澜："好。无论抽到什么，都只是一个入口。"（直进今日一占）

### 12.2 是否需要弹窗

**不需要。** 用首页对话流更自然。星钥塔罗的气质是"陪伴"，弹窗是"告知"，两者冲突。星澜的话本身就是引导，不需要额外浮层打断。唯一例外：免责声明可保留底部固定小字（已存在），不弹窗。

### 12.3 如何解释"开始今日一占"

星澜轻声：
> "今天的牌，已经在等你了。不是要预测什么，只是陪你看看此刻。"

**不解释**抽牌机制、牌阵原理。只给情绪承接 + 边界（"不是预测"）。

### 12.4 如何解释"抽牌不是预测未来"

- 不在首页说，在**首次抽牌前**说（情绪承接位）：
  > "这张牌不是答案，是镜子。映的是现在的你，不是未来的命。"
- 不重复说。`hasSeenSingleGuide = true` 后不再主动提，除非用户在结果页显式问。

### 12.5 如何解释"结果仅供娱乐与自我探索参考"

- **不弹窗**，不单独成步。
- 融在星澜结果页首句话里（首次）：
  > "这只是个入口。觉得对就留一会儿，觉得不对也没关系。"
- 底部保留固定小字免责声明（已有）。
- 首次显示后 `hasSeenDisclaimerHint = true`，星澜不再主动提，小字常驻。

### 12.6 如何提示用户可以点击星澜获得陪伴文案

- 首页首次引导时，星澜说完"你来了"后，**不显式说"你可以点我"**。
- 用极淡的视觉提示：星澜形象有一次极轻的呼吸光晕（动效，非文字）。
- 若用户 8 秒未操作，星澜补一句：
  > "想说话的时候，点我就好。"
- 这条提示只在 `!hasSeenHomeGuide` 时出现一次。

### 12.7 如何提示用户可以查看最近记录

- 不在首页提示。
- 首次进入记录页时，星澜一句：
  > "这里留着之前的自己。想看就看，不想看也没关系。"
- 不强调"回顾""反思"，避免说教感。

### 12.8 如何避免教程太重

| 原则 | 做法 |
|---|---|
| 步数克制 | 全 App 引导总计不超过 6 句星澜话，每场景 1 句 |
| 不阻断 | 每句引导都允许"跳过"或"直接抽牌"，引导不挡主流程 |
| 不重复 | 每个标志位只触发一次，不反复教 |
| 不浮层 | 不用 milk 式遮罩高亮，用对话流 |
| 不解释机制 | 不讲牌阵原理/算法，只给情绪承接 |
| 不游戏化 | 无"完成教程"奖励、无"新手村" |

### 12.9 如何让用户可以跳过

- 每个引导步底部有一个极淡的"跳过"文字按钮（非醒目按钮）。
- 跳过后 `hasSeenXxxGuide = true`，不再触发。
- 跳过不代表"未完成"，只是"用户不需要"。

### 12.10 如何让用户之后还能重新查看

- 设置页加一个入口："重新听听星澜的介绍"。
- 触发后：重置一组 `hasSeenXxxGuide = false`，回到首页由星澜重新引导。
- **不做** milk 的"重放 22 步浮层教程"，而是重放星澜对话流。

### 12.11 气质校验

| 校验项 | 本方案是否符合 |
|---|---|
| 温柔 | ✅ 星澜口吻，短句 |
| 克制 | ✅ 不解释、不重复、不浮层 |
| 不说教 | ✅ 只给情绪承接，不给定义 |
| 不游戏化过重 | ✅ 无奖励、无新手村 |
| 不强迫完成教程 | ✅ 每步可跳过 |
| 不当心理治疗工具 | ✅ 不诊断、不建议 |
| 不当命运预测 | ✅ 反复强调"不是答案，是镜子" |

## 13. 最终结论

### 13.1 最值得借鉴的 5 个点

1. **拟真时序制造陪伴感**（simulateReply 的 typing + delay + 已读）：星澜点击后加"正在输入"+短延迟，离线即可获得强陪伴感，这是 milk 最核心、最可复用的手法。
2. **文案池随机抽取**（customReplies 池随机）：星澜文案库按场景分组随机，每次点击不重复感，无需任何 AI。
3. **tourSteps 数据结构**（{title,content,position,onBefore}）：结构清晰，可迁移为 ArkTS `interface GuideStep`，但执行方式从"浮层"改为"对话流"。
4. **标志位持久化**（tour_seen + 重放按钮）：每个引导场景一个 `hasSeenXxxGuide`，PreferencesHelper 存储，设置页可重置重看。
5. **loading 诗化进度文案 + 启动序列分层**（loading→门禁→教程→问候）：分层启动思路可借鉴，但每层大幅精简。

### 13.2 最需要避开的 5 个点

1. **22 步强制浮层教程**：过长、遮挡、打断，工具型 App 的大忌。
2. **"运势占卜"框架**：周运/日运框架违反星钥塔罗"不预言"底线，塔罗只能做"映照"。
3. **partner 随机心情伪装成真实情绪回应**：不诚实，星澜不模拟情绪，只接住。
4. **恋爱化文案语料**（命运红线/灵魂共振/拍一拍撒娇）：违反星钥塔罗克制气质。
5. **全局可变 let 状态 + DOMElements 对象字面量 + 裸 string[] 文案**：不符合 ArkTS 严格类型，必须显式 interface 化。

### 13.3 对「星钥塔罗」最稳妥的 3 个后续优化方向

**方向一：给星澜点击加"拟真时序"（最高性价比）**
- 改动范围小（仅首页星澜点击响应）。
- 加"星澜正在输入"+0.8~1.5s 延迟，文案从 HOME_TAP 池随机。
- 立刻显著提升陪伴感，离线即可，无任何风险。
- 不碰路由/抽牌/历史/隐私门禁。

**方向二：用对话流替换/补充首次引导（中等性价比）**
- 新增 `OnboardingFlags`（PreferencesHelper）+ `GuideStep[]`（内置文案）。
