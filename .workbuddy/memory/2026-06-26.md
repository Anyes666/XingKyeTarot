## EmotionStarPage 底部按钮 BugFix（软件团队 SOP）

### 问题
EmotionStarPage 底部「← 回到结果页」按钮点击无响应。根因：Button 完全没绑定 `.onClick`。

### 修复
- 文件：`entry/src/main/ets/pages/EmotionStarPage.ets`（唯一改动文件）
- 新增 `private backToResultPage(): void`：`router.back()` + try/catch，失败 fallback `router.replaceUrl({url: Routes.HOME})`。逻辑与左上角 `AppBackButton.handleBack` 等价。
- 给底部 Button 补 `.onClick(() => { this.backToResultPage(); })`，样式属性全部保留。

### 教训（重要）
1. 工程师寇豆码初版自作主张加了 `if (router.getLength() <= 1)` 前置检查，但当前 SDK 里 `router.getLength()` 返回 **string**，不能和 number 比较 → 编译 ERROR。工程师报告"成功"但实际未真实跑通编译，违反"绝不假成功"铁律。
2. QA 严过关静态审查未发现该类型错误（因为没跑编译）。**主理人亲自跑 assembleHap 才抓到**。
3. 修复方式：移除 `getLength()` 检查，让方法与 AppBackButton 完全等价（直接 back + catch fallback）。

## 服务卡片入口隐私门禁守卫专项（合规高优先级 BugFix）

### 问题
EntryAbility.onWindowStageCreate / onNewWant 在收到 targetPage==='dailyFortune' 时直接 loadContent('pages/DailyFortunePage')，**完全没检查 privacyAccepted**，导致未同意协议的用户从桌面卡片点击可直接进入今日星钥页，绕过隐私门禁。合规风险。

### 修复（单文件，直接改，避免 agent abort）
文件：`entry/src/main/ets/entryability/EntryAbility.ets`
1. 新增 import AppStateStore
2. onWindowStageCreate：在 targetPage 分发之前先读 AppStateStore.isPrivacyAccepted(context)，false 时强制 loadContent('pages/LaunchPage') 并 return，忽略卡片 targetPage
3. onNewWant：改为 async，同样加 privacyAccepted 守卫，false 时 loadContent('pages/LaunchPage') 并 return
4. 已同意协议时保留原 targetPage 分发逻辑不变

### 守卫规则
- 冷启动（onWindowStageCreate）：守卫在前，targetPage 分发在后
- 热启动（onNewWant）：同上
- 未同意 → 永远先进 LaunchPage，无论 targetPage 是什么
- 已同意 → 按原逻辑（dailyFortune → DailyFortunePage / 其他 → LaunchPage 走 Onboarding/firstExperience 分流）
- 复用 AppStateStore.isPrivacyAccepted（已有方法，不新增 Store）
- 不引入 pendingTarget 复杂链路（spec 不要求）
